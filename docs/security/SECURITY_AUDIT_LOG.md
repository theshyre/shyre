| SAL-006 | 2026-04-16 | Medium (confidentiality) | `time_entries_select` used `user_has_team_access(team_id)` as its middle clause, which meant any team member could `SELECT` any other team member's time entries — including duration, billable flag, description, and (via joined `projects` / `customers`) any rate / client detail. Write-side policies (INSERT / UPDATE / DELETE) were already tight (own-entry or owner/admin) since migration 009, but the read-side was inherited from the initial multi-tenant policy and never tightened. The user's design intent is "members log their own time and nothing else" — this policy violated that default. During an audit on 2026-04-16 I initially flagged three RLS gaps (also `user_settings` and `projects` / `customers` writes) but verification against `pg_policies` on the live DB showed those had already been resolved in migrations 004 and 009 respectively. This SAL is the one remaining real gap. | Resolved | First-attempt migration `20260416220000_sal006_tighten_time_entries_select.sql` replaced the middle clause with `user_team_role(team_id) IN ('owner', 'admin')` and preserved the existing third clause `user_can_see_cross_team_entries(p.customer_id)`. Integration tests caught a subtle bug: `user_can_see_cross_team_entries`'s FIRST branch returns TRUE for any member of the customer's primary team, silently re-opening the same-team read gate that the tightened middle clause closed — so a plain member could still read another member's entry via clause 3. Correction migration `20260416231353_sal006_correction_inline_cross_team_branches.sql` inlines the three cross-team branches explicitly (owner/admin of the customer's primary team; member of a team with `customer_shares.can_see_others_entries = true`; customer-admin permission) so same-team members no longer reach the cross-team gate by accident. Integration tests in `src/__integration__/rls/time-entries-rls.test.ts` under `describe("SAL-006: same-team member visibility is tight by default")` exercise three cases: (a) same-team member cannot read the owner's entry, (b) member can still read their own entry, (c) owner can read a member's entry. Phase 3 of `docs/reference/rate-and-access-plan.md` will add a `time_entries_visibility` config (own_only / read_all / read_write_all) on team_settings + projects that an owner can opt into for teams that want looser defaults. Lesson: a helper that looks relevant to your rule may encode other rules you don't want — prefer inlining small branches over delegating to a generic-sounding helper when the policy's tightness hinges on each branch being explicitly narrow. |
